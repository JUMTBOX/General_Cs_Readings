## 파티셔닝(scale-up) VS 샤딩(scale-out)

**문제 상황: 데이터 폭증의 위기**
- 단일 테이블, 단일 DB 구조가 한계에 도달
  - `I/O 병목 심화`: 데이터 증가로 인덱스 길이 증가, 디스크 접근 비용 상승
  - `동시성 충돌 증가`: 트랜잭션 경합 및 락 대기 시간 증가
  - `운영 비용 증가`: 백업, 복구, 배포 시간이 기하급수적으로 증가
  - `물리적 한계 도달`: CPU, 메모리, 디스크 확장의 한계

**Solution**
```text
💡데이터를 논리적으로 또는 물리적으로 나누어야 한다.
```

**파티셔닝(Partitioning)**
- `한 지붕 아래 여러 방 만들기`
- `단일 DB 서버` 내부에서 하나의 거대한 테이블을 여러 파티션(저장 단위)으로 나누어 저장
- 개발자는 하나의 테이블 처럼 사용하지만, DB 내부에서는 여러 파티션으로 분리되어 저장
- `DB 엔진 수준에서 관리하여준다.`

**파티셔닝의 종류**
1. Range (범위)
   - 날짜나 숫자 범위를 기준으로 분할 (ex: 월별 로그, 연도별 데이터) 
2. List (목록)
   - 특정 값의 목록을 기준으로 분할 (ex: 지역코드- 서울, 부산)
3. Hash (해시)
   - 해시 함수를 통해 데이터를 균등하게 분산 (특정 값의 쏠림을 방지하기 위한 전략)
4. Composite (복합)
   - 두 개 이상의 파티션 전략을 조합 (ex: Range + Hash)

**샤딩(Sharding)**
- `옆 동네에 새 집 짓기`
- 데이터를 `서로 다른 물리적 서버(인스턴스)`들에 분산 저장
- 각 서버를 `샤드(shad)`라고 부르며, 수평적 확장의 핵심 기술

**차이점**
1. 물리적 위치가 다르다.
    - 파티셔닝은 `같은 DB 인스턴스` 안에서 분리, 샤딩은 `서로 다른 DB인스턴스`로의 분산
2. 확장 방향이 다르다.
    - 파티셔닝은 단일 서버 안에서의 최적화, 샤딩은 여러 서버로의 확장 전략 
3. 💡책임 주체가 다르다.💡
    - 파티셔닝은 `DB 엔진`이, 샤딩은 대개 `어플리케이션`(또는 미들웨어)이 어디로 보낼지 결정한다. (샤딩은 Shard key 기반 라우팅 필요)

## 실무 활용

**파티셔닝**
- `로그 테이블`: 시간 기준으로 지속적으로 누적되는 대용량 테이블
- `날짜 기반 데이터`: 최신 데이터 조회 성능이 중요할 때 (조회 조건에 날짜 컬럼이 자주 사용될 때)
- `대량 데이터 정리 효율`: 오래된 데이터를 통째로 정리하고 싶을 때

**샤딩**
- `대규모 트래픽/ 데이터 규모`: 단일 DB 한계를 넘을 때
- `글로벌 서비스`: 한국 서버, 미국 서버 등 지역별 데이터 물리 분산 (데이터의 locality 확보)
- `성능 극한(scale-up limit)에 직면 했을 때`: 단일 서버 사양을 아무리 높여도 성능이 나오지 않을 때

`샤딩을 하면 Join 연산이 매우 힘들어진다. (데이터 모델링 중요)`
`샤딩은 선택이 아니라 최후의 수단에 가깝다. 단일 DB의 한계를 넘었을 때 도입 고려`
